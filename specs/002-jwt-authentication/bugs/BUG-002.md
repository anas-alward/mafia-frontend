# Bug Report: API Client Should Be Project-Wide Shared Utility, Not Auth-Scoped

**Type**: Spec gap
**Severity**: Medium
**Feature**: 002-jwt-authentication
**Reported**: 2026-06-26
**Status**: Patched
**Patched**: 2026-06-26

## Description

The API client module (`src/features/auth/api/client.ts`) was implemented as an auth-specific module. It contains:

1. The base `request<T>()` helper — a generic fetch wrapper with `API_BASE` resolution, `credentials: 'include'`, JSON content-type, and error handling
2. Auth-specific endpoint functions (`signUp`, `login`, `logout`, `refresh`, `getMe`, `forgotPassword`, `resetPassword`, `changePassword`)
3. Auth-specific types (`UserDto`, `AuthTokens`, `SignUpRequest`, etc.)

The `request<T>()` helper and `API_BASE` configuration are not auth-specific — they are project-wide infrastructure that every feature (game data, leaderboards, user profiles, etc.) will need when calling the Django backend. By embedding them inside the auth feature, other features are forced to either import from `#/features/auth/api/client` (making auth a dependency of unrelated features) or duplicate the fetch wrapper.

## Root Cause Analysis

The plan was written from the perspective of "the auth feature needs an API client." The Architecture section places `api/client.ts` under `src/features/auth/` as a natural extension of the feature folder pattern. However, this didn't account for the fact that:

1. **`credentials: 'include'` is project-wide**: Every Django API call needs to forward the refresh cookie, not just auth endpoints
2. **`API_BASE` is project-wide**: There is one Django backend serving all API endpoints, not one per feature
3. **`request<T>()` is project-wide**: Every feature will need typed fetch wrappers with consistent error handling

The feature-folder pattern (`features/auth/`, `features/game/`, etc.) is correct for domain logic but not for cross-cutting infrastructure. The shared fetch wrapper belongs in `src/lib/` alongside other project utilities.
