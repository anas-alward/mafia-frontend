# Bug Report: Implementation Assumed Self-Contained Server-Side Auth — Should Be Django API Client

**Type**: Spec gap
**Severity**: Critical
**Feature**: 002-jwt-authentication
**Reported**: 2026-06-26
**Status**: Patched
**Patched**: 2026-06-26

## Description

The entire implementation assumes authentication logic (JWT creation/verification, password hashing, user storage) runs inside Cloudflare Workers using D1. In reality, the user has a Django backend that already handles all of this. The frontend should be a **client** that calls Django's auth API endpoints, not a self-contained full-stack auth system.

The following were implemented but should NOT exist:
- Server-side JWT creation/verification (`jose`, `src/features/auth/server/auth.ts`)
- Server-side password hashing (PBKDF2 via Web Crypto)
- D1 database schema and queries (`schema.sql`, `src/features/auth/server/db.ts`)
- Server-side cookie handling (`src/features/auth/server/cookies.ts`)
- `createServerFn` handlers for auth operations (should be `fetch()` calls to Django)

## Root Cause Analysis

The spec was written generically — it says "the server MUST hash passwords" and "the server MUST issue JWTs" — without specifying WHICH server (Cloudflare Workers vs Django). The `Assumptions` section never documented that a Django backend already exists and handles user management. The clarifications session asked about rate limiting ("handled by backend") but never probed whether the backend handles ALL auth or just rate limiting.

During `/speckit-plan`, the assumption was that auth runs inside Workers because:
1. TanStack Start supports `createServerFn` natively
2. Cloudflare D1 is the project's database
3. The spec never said "call an external API"

The architecture decision was reasonable given the spec, but the spec itself was incomplete — it lacked the critical context that a Django backend exists.
